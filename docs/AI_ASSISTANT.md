# ТЗ: ИИ-ассистент для репортов и алертов

## 1. Цель

Прототип чат-бота: пользователь описывает желаемый репорт или алерт свободным
текстом («покажи расход топлива по заправкам за неделю», «алерт: если машина
перегрелась — письмо механику, но не чаще раза в час»), ИИ возвращает **готовый
валидный конфиг** в формате REPORTS.md §4 / ALERTS.md §3, редактор его применяет.

Это возможно ровно потому, что конфигураторы декларативные: ИИ не пишет ни SQL,
ни кода — только JSON, который проходит те же серверные валидаторы, что и ввод
из UI.

## 2. Стек

- **Официальный PHP SDK Anthropic**: `composer require anthropic-ai/sdk` —
  единственная новая зависимость.
- Модель: **`claude-opus-4-8`** (константа в конфиге сервиса).
- **Structured outputs** (`outputConfig` с `json_schema`) — API гарантирует,
  что ответ соответствует схеме; никакого парсинга markdown-заборов.
- Ключ: `ANTHROPIC_API_KEY` в `app/.env.local` (не коммитится). **Без ключа
  проект полностью работоспособен**: ИИ-эндпоинты отвечают 503, чат-панели в UI
  скрыты (контроллер передаёт флаг `ai_enabled` в шаблоны).

## 3. Архитектура

Один сервис `App\Ai\AiConfigurator` с двумя публичными методами:

```php
/** @return array{reply: string, config: array} */
public function generateReport(string $prompt, array $history, ?array $currentConfig): array;
public function generateAlert(string $prompt, array $history, ?array $currentConfig): array;
```

Внутри оба метода делают одно и то же, отличаются только «профилем»
(system prompt + JSON-схема + валидатор):

1. Собрать messages: история чата + текущий конфиг (если редактируется
   существующий) + новый prompt.
2. Вызвать API:

```php
$message = $client->messages->create(
    model: 'claude-opus-4-8',
    maxTokens: 16000,
    thinking: ['type' => 'adaptive'],
    system: [[
        'type' => 'text',
        'text' => $systemPrompt,                       // см. §6, стабильный
        'cacheControl' => ['type' => 'ephemeral'],     // промпт-кэш: system большой и не меняется
    ]],
    messages: $messages,
    outputConfig: ['format' => [
        'type' => 'json_schema',
        'schema' => $responseSchema,                   // {reply: string, config: {...}}
    ]],
);
// первый text-блок гарантированно содержит валидный JSON по схеме
```

3. Прогнать `config` через **тот же валидатор**, что и UI
   (`ReportValidator` / `AlertValidator`; для алертов дополнительно построить
   `summary`).
4. Если невалиден — **один** повторный вызов: к messages добавляются ответ
   модели и user-сообщение «Конфигурация не прошла валидацию: <список ошибок>.
   Исправь и верни конфиг заново». Снова невалиден → HTTP 422 с ошибками
   (пользователь видит их в чате).
5. Вернуть `{reply, config}`; для алертов — ещё и `summary`.

Схема ответа (`$responseSchema`) — оба поля обязательны:

```json
{
  "type": "object",
  "properties": {
    "reply": {"type": "string", "description": "Короткий ответ пользователю по-русски: что сделано и какие допущения приняты"},
    "config": {"...": "JSON-схема конфига репорта или алерта"}
  },
  "required": ["reply", "config"],
  "additionalProperties": false
}
```

Вложенную схему `config` описать настолько строго, насколько позволяет
structured outputs (enum для fn/op/viz/type узлов, `additionalProperties: false`);
тонкие правила (совместимость viz↔query, связность графа) всё равно ловит
серверный валидатор — двухуровневая защита.

## 4. API

| Метод и путь | Тело | Ответ |
|---|---|---|
| `POST /api/ai/report` | `{"prompt": str, "history": [{role, content}], "current_config": obj\|null}` | `{"reply": str, "config": obj}` |
| `POST /api/ai/alert` | то же | `{"reply": str, "config": obj, "summary": str}` |

- `history` — предыдущие сообщения чата (без system), хранится в JS на клиенте;
  сервер полностью stateless.
- 503 если `ANTHROPIC_API_KEY` пуст; 422 если ИИ дважды вернул невалидный
  конфиг; ошибки Anthropic API (429/5xx) — пробросить как 502 с текстом.
- Таймаут HTTP-клиента ≥ 120с: генерация с thinking может занимать десятки секунд.

## 5. UI: чат-панель

Общий компонент (один кусок JS) на страницах `/report/{id}/edit` и
`/alert/{id}/edit`, правая колонка:

- лента сообщений (user / assistant), поле ввода, кнопка «Отправить», спиннер;
- после ответа: `reply` появляется в ленте, `config` **сразу применяется**
  в редактор (форма+предпросмотр репорта / канвас алерта) — пользователь видит
  результат и может продолжить диалог: «а теперь по дням, не по часам» —
  с каждым запросом уходит `current_config`, поэтому ИИ правит, а не пересоздаёт;
- применённый конфиг НЕ сохраняется автоматически — пользователь жмёт
  «Сохранить» сам;
- кнопка «Создать с ИИ» на списках `/reports`, `/alerts` открывает редактор
  нового объекта с фокусом в чате.

## 6. System prompt (скелет, хранить в `app/prompts/report.md` и `alert.md`)

Собирается конкатенацией статичных файлов и генерируемых блоков —
**порядок стабилен**, чтобы работал промпт-кэш:

1. Роль и задача: «Ты конфигуратор репортов системы телеметрии логистического
   автотранспорта. По запросу пользователя на естественном языке верни JSON…»;
2. Словарь данных: описание таблицы `events`, все поля из `Schema` с
   пояснениями и **значениями измерений** (типы событий, регионы, диапазоны id
   машин/водителей/заправок — из DATA_MODEL.md §2);
3. Полное описание формата конфига (выжимка из REPORTS.md §4 / ALERTS.md §3–4;
   для алертов — каталог узлов, сгенерированный из `NodeCatalog`);
4. Правила: «только поля из словаря», «если период не указан — последние 7
   дней», «если канал уведомления не указан — email», «допущения перечисляй
   в reply», «отвечай по-русски»;
5. Few-shot: 2–3 пары «запрос → конфиг» из `docs/examples/`.

Динамика (текущий конфиг, запрос) — только в `messages`, никогда в system.

## 7. Безопасность и стоимость

- Конфиг от ИИ — недоверенный ввод: та же валидация по белым спискам, что и у
  человека; в SQL попадает только через `QueryBuilder`.
- Ключ API живёт на сервере; браузер ходит только в наши эндпоинты.
- Проект без авторизации ⇒ эндпоинты ИИ доступны всем, кто видит порт: не
  выставлять наружу; для учебного докера это приемлемо (задокументировать).
- Один запрос ≈ 5–15k входных токенов (system кэшируется, повторные — дёшевы);
  учесть в README, что фича платная и опциональная.

## 8. Критерии приёмки

- «Покажи средний расход топлива по заправкам за последнюю неделю» → валидный
  репорт: bar, avg(fuel_consumption_l100), group by last_fuel_station_id,
  last_hours 168 — рендерится в предпросмотре без правок.
- «Дашборд для механика: перегревы, давление в шинах и топ проблемных машин» →
  репорт из ≥3 виджетов, все проходят валидацию.
- «Алерт: если машина перегрелась выше 105 градусов — письмо механику, не чаще
  раза в час» → граф threshold → dedup → email, валиден, summary осмыслен.
- Диалог из 2 шагов: создать репорт → «замени график по часам на график по
  дням» → ИИ меняет только `time_bucket` существующего конфига.
- Без `ANTHROPIC_API_KEY`: панели скрыты, эндпоинты 503, остальной проект
  работает как раньше.
- Невалидный ответ ИИ (сымитировать сломанной схемой) → после ретрая 422 с
  внятной ошибкой в чате, редактор не ломается.
